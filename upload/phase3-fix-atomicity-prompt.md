# طلب إصلاح — POS Sale Atomicity (قبل Phase 4)

## المشكلة
`posSale()` الحالية بتنفّذ `createInvoice` → `postInvoice` → `recordSale` (لكل سطر) كخطوات منفصلة، كل واحدة بـ transaction داخلية مستقلة — مش transaction واحدة شاملة كما نص `modules/pos.md`. لو فشلت خطوة بعد نجاح خطوة سابقة (مثلًا `recordSale` فشل بعد `postInvoice`)، بتنتج حالة جزئية: فاتورة مرحّلة بإيراد مسجّل بدون COGS/خصم مخزون مقابل.

## المطلوب
أعد بناء `posSale()` لتنفّذ **كل** الخطوات (إنشاء الفاتورة، ترحيلها، تسجيل كل حركات البيع) داخل **transaction واحدة `db.$transaction()`**:

1. استخدم نفس نمط `createJournalEntryOn(tx, ...)` الموجود من قبل — أي دالة من الخدمات الحالية (`createInvoice`, `postInvoice`, `recordSale`) تحتاج تتحول لنسخة تقبل `tx` اختياري (زي `recordMovement(input, tx?)` الموجودة فعليًا في stock-movement.service.ts)، **بدون** تكرار منطقها — فقط تمرير `tx` بدل استخدام `db` مباشرة داخلها
2. **تذكير حرج (قاعدة Phase 1)**: أي استعلام داخل `tx` يحتاج `tenantId` صريح في where/data — مفيش middleware عزل تلقائي جوه transaction client
3. لو أي خطوة فشلت (بما فيها فحص المخزون الداخلي في `recordSale`)، الـ transaction بالكامل تتراجع (rollback) — مفيش فاتورة، مفيش قيد، مفيش حركة مخزون
4. أضف اختبار رابع في `/api/tests`: **محاكاة فشل جزئي متعمد** (مثلًا منتج بمعرف غير موجود في السطر الثاني من السلة) → تحقق إن الفاتورة اللي كانت هتُنشأ من السطر الأول **لم تُنشأ** أصلًا (لا نتيجة جزئية حتى لو السطر الأول كان سليمًا)

## المطلوب في الرد
1. الكود المُعدَّل لـ `posSale()` وأي دالة احتاجت تعديل لدعم `tx`
2. نتيجة الاختبار الرابع (الفشل الجزئي المتعمد) فعليًا
3. تأكيد إن الاختبارات 1-11 الموجودة لسه بتنجح بعد التعديل
