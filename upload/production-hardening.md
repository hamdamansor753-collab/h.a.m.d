# modules/production-hardening.md — Production Hardening

## الهدف
حل الملاحظات الأربعة المتراكمة في `review/deferred-items-log.md` + الانتقال الفعلي من SQLite (بيئة تطوير) لـ PostgreSQL (بيئة إنتاج) مع تفعيل RLS الحقيقي المكتوب من Phase 0 والتحقق منه فعليًا. هذه المرحلة **لا تضيف ميزات جديدة** — هدفها الوحيد: تحويل الـ prototype المُختبَر أمنيًا لنظام جاهز للاستخدام الإنتاجي الأول.

---

## البند 1: الترحيل الفعلي لـ PostgreSQL + تفعيل RLS

**المطلوب**:
1. تغيير `provider` في `schema.prisma` من `sqlite` لـ `postgresql`
2. تشغيل الـ migration الأساسي على قاعدة PostgreSQL فعلية (Docker container محلي كافٍ للتحقق)
3. **تنفيذ ملف `prisma/sql/tenant_rls_postgres.sql`** فعليًا على القاعدة (مش بس وجوده كملف — تشغيله والتحقق إنه اتنفذ بدون أخطاء)
4. اختبار حاسم: تعطيل middleware العزل في `db.ts` **مؤقتًا للاختبار فقط** ومحاولة query عابر لـ tenant — يجب أن **يُرفض من RLS نفسه على مستوى القاعدة**، لا الكود. هذا يثبت العزل حقيقي على مستوى البنية التحتية، مش بس على مستوى التطبيق (دفاع من طبقتين حقيقي)
5. إعادة تفعيل middleware، وتوثيق نتيجة الاختبار في worklog

## البند 2: ترقيم تسلسلي مقفول (يحل الملاحظتين 1 و 2)

**المشكلة**: `nextInvoiceNumber` و `nextPurchaseOrderNumber` بيعتمدوا على `count()+1` — عرضة للتعارض تحت التزامن.

**الحل الموحّد**: جدول تسلسل مركزي:
```prisma
model SequenceCounter {
  tenantId String
  sequenceKey String // "invoice" | "purchase_order" | ...
  lastValue   Int    @default(0)

  @@id([tenantId, sequenceKey])
}
```
دالة واحدة `getNextSequenceValue(tx, tenantId, sequenceKey)` تستخدم `UPDATE ... SET lastValue = lastValue + 1 ... RETURNING lastValue` (atomic، لا race) بدل `count()`. تُستخدم في **كل** مكان فيه ترقيم تسلسلي حاليًا (فواتير، أوامر شراء) — استبدال، لا إضافة منطق مواز.

## البند 3: قفل صف لفحص كفاية المخزون (يحل الملاحظة 3)

**المشكلة**: `recordMovement` بيعمل check-then-write بدون قفل صريح.

**الحل**: استبدال الفحص بعملية atomic واحدة على مستوى القاعدة:
```sql
UPDATE "StockLevel" SET quantity = quantity - $1
WHERE productId = $2 AND warehouseId = $3 AND quantity >= $1
RETURNING quantity;
```
لو النتيجة صف فارغ (0 rows)، يعني الكمية مش كافية → رفض العملية. هذا يلغي الفجوة الزمنية بين "القراءة" و"الكتابة" تمامًا، مش بس يقلل احتمالها.

## البند 4: دقة ضريبة الرواتب المصرية (يحل الملاحظة 4)

> **تنبيه مهم**: الأرقام دي محدّثة حسب قانون 7/2024 وتعديلاته كما تُطبَّق فعليًا في 2026، لكن **لازم تتأكد منها مع محاسب قانوني قبل استخدامها في رواتب حقيقية** — القوانين الضريبية بتتغير بقرارات وزارية، ومصادرها بتختلف شوية في تفاصيل الإعفاء الشخصي بالضبط.

**شرائح ضريبة الدخل السنوية (تصاعدية، 7 شرائح)**:
| الشريحة (سنوي) | النسبة |
|---|---|
| حتى 40,000 جنيه | 0% (معفى) |
| 40,001 – 55,000 | 10% |
| 55,001 – 70,000 | 15% |
| 70,001 – 200,000 | 20% |
| 200,001 – 400,000 | 22.5% |
| 400,001 – 1,200,000 | 25% |
| فوق 1,200,000 | 27.5% |

**التأمينات الاجتماعية** (على الأجر التأميني = الراتب الأساسي فقط، بدون البدلات):
- حصة الموظف: 11%
- حصة صاحب العمل: 19%
- **حد أقصى للوعاء التأميني الشهري: 10,500 جنيه** → أقصى خصم تأمينات شهري للموظف: 1,155 جنيه (وليس نسبة غير محدودة كما في التنفيذ الحالي المبسّط)

**المطلوب في الكود**:
1. `EgyptPayrollRuleProvider` يُعاد بناؤه بشرائح ضريبية سنوية فعلية (تحويل الراتب الشهري لسنوي، توزيع على الشرائح، قسمة الضريبة السنوية /12 للخصم الشهري — هذا التفصيل مهم: الضريبة تُحسب سنويًا وتُوزَّع شهريًا، مش شريحة شهرية منفصلة)
2. الأرقام (حدود الشرائح، النسب، حد أقصى التأمين) تُخزَّن في **ثوابت منفصلة قابلة للتعديل بسهولة** (ملف config أو جدول)، **مش hardcoded داخل منطق الحساب** — عشان أي تحديث تشريعي لاحق يبقى تعديل رقم، مش إعادة كتابة كود
3. تعليق واضح في الكود يوجّه المستخدم للتحقق من الأرقام مع محاسب قبل الاستخدام الفعلي

## معيار الإتمام الكلي
- تشغيل فعلي على PostgreSQL + إثبات RLS يعمل حتى مع تعطيل middleware التطبيق
- كل الاختبارات الـ22 السابقة PASS بعد الترحيل لـ PostgreSQL (لا افتراض إنها هتفضل شغالة)
- اختبار تزامن جديد: محاولتين بيع متزامنتين لنفس المنتج بكمية تتجاوز المتاح مجتمعة → واحدة تنجح والتانية تُرفض (لا سباق)
- اختبار حساب ضريبة صحيح لراتب عيّنة (مقارنة يدوية بالشرائح الجديدة)
