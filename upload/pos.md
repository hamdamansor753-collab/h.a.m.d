# modules/pos.md — POS (Phase 3)

## يعتمد على
Core (Phase 0) + Accounting (Phase 1) + Inventory (Phase 2) — الثلاثة مكتملين ومُختبرين.

## النطاق
واجهة بيع سريعة (نقطة بيع) تنتج **فاتورة مُرحَّلة تلقائيًا** + **حركة مخزون خارجة** + **قيد COGS** — في عملية واحدة متكاملة، بدل ما البائع يعمل 3 خطوات منفصلة.

## المبدأ الأساسي: POS لا يخترع منطق جديد
موديول POS **لا ينشئ** جدول بيانات جديد للمبيعات، ولا منطق محاسبي جديد. هو **طبقة واجهة (UI + orchestration)** فوق الخدمات الموجودة فعليًا:
- إنشاء + ترحيل الفاتورة → `invoice.service.ts` من Phase 1 (بدون تعديل)
- خصم المخزون + قيد COGS → `sales-movement.service.ts` من Phase 2 (بدون تعديل)

السبب: قاعدة "مصدر واحد للحقيقة" — لو POS عمل حساباته الخاصة، هيبقى فيه احتمال تضارب بينه وبين الفواتير العادية.

## سيناريو البيع (Sale Flow)
دالة تنسيقية جديدة واحدة فقط: `posSale()` في `src/modules/pos/pos-sale.service.ts`:

```
posSale(input: { warehouseId, customerName, lines: [{ productId, quantity, unitPrice }] }):
  1. يبدأ transaction واحدة تلف كل الخطوات (atomicity: البيع كله يحصل أو لا شيء)
  2. لكل سطر: يتحقق إن الكمية متاحة (عبر stock-movement.service) — لو أي سطر غير كافٍ، تُرفض العملية كاملة قبل أي كتابة
  3. ينشئ Invoice (DRAFT) عبر invoice.service منطق موجود، بنفس الأسطر
  4. يرحّل الفاتورة فورًا (postInvoice) → قيد مبيعات (AR/Cash مدين، Revenue+Tax دائن)
  5. لكل سطر: يستدعي recordSale (من sales-movement.service) → StockMovement(SALE) + قيد COGS (مدين COGS، دائن Inventory)
  6. يرجّع: الفاتورة المرحّلة + إجمالي COGS + صافي الربح التقريبي للعملية (Revenue - COGS)
```

## القاعدة الإلزامية: طريقة الدفع
Phase 3 نطاق مبسّط: **نقدي فقط (Cash)** — كل عملية POS تفرض `paymentMethod = CASH`، وتربط مباشرة بحساب "الخزينة/النقدية" (Cash) بدل "حسابات مدينون" (AR) المستخدم في الفواتير الآجلة العادية. البيع بالتقسيط/الأجل مؤجَّل لموديول لاحق (Treasury/POS متقدم).

## نموذج البيانات
**لا جداول جديدة مطلوبة فعليًا** إلا حقل واحد اختياري:
```prisma
// إضافة على Invoice الموجودة من Phase 1 (لا تعديل جوهري في البنية)
model Invoice {
  // ... الحقول الموجودة
  channel  InvoiceChannel @default(MANUAL) // "MANUAL" | "POS"
}

enum InvoiceChannel {
  MANUAL
  POS
}
```
ده بيسمح بتمييز فواتير POS عن الفواتير العادية في التقارير، بدون أي تكرار في البيانات.

## واجهة UI (POS Screen)
- شاشة مبسّطة، أزرار كبيرة، بحث سريع عن منتج بالاسم/SKU
- سلة (Cart) في الجانب، إجمالي مباشر مع الضريبة
- زر واحد "إتمام البيع" يُنفّذ `posSale()` بالكامل
- بعد نجاح البيع: طباعة/عرض إيصال مبسّط (لا يحتاج تصميم فاتورة رسمية كاملة الآن)

## الصلاحيات
صلاحية جديدة واحدة: `pos:sell` — تُمنح لدور جديد `cashier` (أضعف من accountant، بدون وصول لتقارير أو إعدادات)، بالإضافة لـ admin.

## معيار الإتمام
- بيع منتج من شاشة POS → التحقق: (أ) فاتورة جديدة بـ `channel=POS` ظاهرة في `/api/invoices` (ب) `StockLevel` نقصت بالكمية الصحيحة (ج) قيدين محاسبيين ظهروا في `/api/journal` (مبيعات + COGS) متوازنين كل واحد على حدة
- محاولة بيع كمية أكبر من المتاح من شاشة POS → رفض قبل أي كتابة، لا فاتورة ولا حركة مخزون تُنشأ (لا "نصف عملية")
- اختبار عزل tenant على عمليات POS
