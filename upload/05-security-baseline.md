# 05 - خط الأساس الأمني (Security Baseline)

> قواعد إلزامية مستخلصة من أخطاء حقيقية طلعت في مشاريع سابقة (Al-Zaeem, H.A.M.D). أي انتهاك لأي قاعدة هنا = رفض تلقائي في المراجعة.

## 1. Tenant Isolation
- [ ] كل query على جدول به `tenantId` يعمل عبر RLS مُفعّلة (لا فلترة يدوية فقط في التطبيق)
- [ ] لا يوجد أي "static cache key" يُستخدم عبر tenants مختلفة (كل مفتاح cache يتضمن `tenantId`)
- [ ] لا dashboard query أو aggregation بدون scope واضح لـ tenant محدد

## 2. API Routes
- [ ] كل API route يستخدم Prisma يحتوي على `export const runtime = 'nodejs'` بشكل صريح
- [ ] كل route يبدأ بفحص المصادقة (auth) ثم الصلاحية (permission) قبل أي منطق آخر
- [ ] كل مدخل من المستخدم يُفحص بـ Zod schema قبل الوصول لطبقة الخدمة (لا استثناءات)

## 3. المصادقة والصلاحيات
- [ ] فحص الصلاحية يتم في **service layer**، لا في الواجهة فقط
- [ ] لا يوجد أي "super-admin backdoor" ضمني أو مخفي (كل صلاحية super-admin موثقة ومفحوصة بوضوح في الكود)

## 4. البيانات الحساسة
- [ ] كلمات المرور: hashing بمعيار قوي (bcrypt/argon2)، لا تشفير عكسي أبدًا
- [ ] لا بيانات حساسة (مفاتيح API، كلمات مرور) في logs أو error messages تُعرض للمستخدم

## 5. الاختبارات
- [ ] كل موديول جديد يحتوي على حد أدنى من اختبارات tenant-isolation (محاولة وصول tenant A لبيانات tenant B يجب أن تفشل، ويُختبر هذا فعليًا لا افتراضيًا)

## 6. معالجة الأخطاء
- [ ] لا `.errors` مع Zod (يجب استخدام `.issues` — تجنبًا لتكرار خطأ سابق مع Zod 4)
- [ ] رسائل الخطأ المعروضة للمستخدم مترجمة عبر i18n، ولا تكشف تفاصيل تقنية داخلية (stack trace, أسماء جداول)
